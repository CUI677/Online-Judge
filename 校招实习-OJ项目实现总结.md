# 在线 OJ 判题系统实现总结

这份文档是对应简历项目介绍写的，重点不是源码逐行解释，而是帮你在面试里把“写了什么”和“怎么实现的”顺着讲出来。

适用场景：

1. 面试官让你介绍 OJ 项目。
2. 面试官追问你简历上的某个技术亮点是怎么做的。
3. 你需要把项目介绍扩展成 3 到 5 分钟的口述版本。

## 一、项目整体是怎么实现的

### 1. 项目整体架构

这个 OJ 项目整体分成两部分：

- `oj_serve`：主业务服务，负责题库展示、题目详情、提交代码、返回判题结果。
- `compile_serve`：编译运行服务，负责真正去编译和执行用户提交的代码。

这样拆分的原因：

- 主服务只关注 Web 请求和业务逻辑。
- 编译执行用户代码属于高风险操作，单独拆出来更清晰。
- 后续如果并发增加，可以部署多台编译机，由主服务做简单分发。

你可以这样说：

> 我把项目拆成了 OJ 主服务和编译服务两部分。OJ 主服务负责题目展示和判题请求入口，编译服务专门负责用户代码的编译运行。这样做一方面职责更清晰，另一方面也方便后续把编译服务独立扩容。

### 2. 一次完整判题请求的流程

完整流程可以按这个顺序讲：

1. 用户在题目页面输入代码并提交。
2. 前端把代码和输入数据通过 HTTP POST 发给 OJ 主服务。
3. OJ 主服务根据题号找到题目配置，包括题目描述、代码模板、CPU 限制、内存限制。
4. 主服务把用户代码拼接到题目的 `header.cpp` 和 `tail.cpp` 中，组装成一份完整可编译代码。
5. 主服务再把代码、输入和资源限制封装成 JSON，请求编译服务。
6. 编译服务收到请求后，生成临时源码文件，调用 `g++` 编译。
7. 编译成功后，子进程执行用户程序，并通过 `setrlimit` 限制 CPU 和内存。
8. 执行结束后，把标准输出、标准错误、运行状态封装成 JSON 返回主服务。
9. 主服务再把结果回传前端展示。

这就是项目最核心的主链路。

## 二、简历亮点是怎么实现的

## 亮点 1：基于 `cpp-httplib` 搭建 OJ 服务端，提供题库列表、题目详情、代码提交判题等 HTTP 接口

### 实现思路

OJ 主服务本质上就是一个 HTTP 服务器，使用 `cpp-httplib` 提供路由接口：

- `GET /` 或 `GET /all_questions`：返回题库首页
- `GET /question/{number}`：返回题目详情页
- `POST /judge/{number}`：接收用户代码并进入判题流程
- `GET /healthz`：健康检查接口

对应实现位置主要在：

- `/root/ONLINEJUDGE/oj_serve/oj_server.cc`

### 你可以怎么讲

> 我用 `cpp-httplib` 搭了一个轻量级 HTTP 服务，通过不同路由处理题库列表、题目详情和代码提交。这样项目对外表现就是一个标准 Web 服务，前端只需要按接口访问就可以。

### 面试官可能追问

#### 为什么用 `cpp-httplib`？

可以答：

- 它是轻量级 C++ HTTP 库，接入简单，适合做这种课程项目和校招项目。
- 我这个项目重点是 OJ 判题流程和 Linux 进程控制，所以没有把重点放在复杂 Web 框架选型上。

#### 为什么不用数据库？

可以答：

- 当前题库规模不大，题目直接从文件系统加载即可。
- 这样实现简单，也方便先把主流程跑通。
- 如果后续题目数量增加，可以再接数据库。

## 亮点 2：设计 `Model + Control + View` 分层结构，分别负责题库加载、请求处理和页面渲染

### 实现思路

这个项目在 OJ 主服务里做了比较清晰的模块划分：

- `Model`
  - 负责加载题库数据
  - 从 `questions.list` 和每道题的 `desc.txt`、`header.cpp`、`tail.cpp` 中读取内容
  - 提供查询所有题目和单题信息的接口
- `Control`
  - 负责处理 HTTP 请求
  - 调用 Model 获取题目
  - 组织判题请求
  - 调用编译服务或本地编译模块
- `View`
  - 负责把题目数据渲染成 HTML 页面
  - 包括题库首页和题目详情页

对应主要文件：

- `/root/ONLINEJUDGE/oj_serve/oj_model.hpp`
- `/root/ONLINEJUDGE/oj_serve/oj_control.hpp`
- `/root/ONLINEJUDGE/oj_serve/oj_view.hpp`

### 这样设计的好处

- 业务层次更清楚。
- 数据管理、请求处理、页面展示互不混杂。
- 后续改题库、改页面、改判题流程时影响范围更小。

### 你可以怎么讲

> 我在主服务里做了一个比较简单的 MVC 分层。Model 负责题库数据加载，Control 负责处理用户请求和判题逻辑，View 负责页面渲染。这样做的好处是代码结构更清晰，后续维护和扩展比较方便。

## 亮点 3：实现独立编译运行服务，接收用户代码后动态生成临时源码文件，通过 `fork + execlp + waitpid` 完成编译与执行流程

### 实现思路

编译服务对外提供一个接口：

- `POST /compile_and_run`

收到请求后会做这些事：

1. 解析前端传来的 JSON，请求里包含代码、输入、CPU 限制、内存限制。
2. 为本次提交生成唯一文件名。
3. 把代码写到临时 `.cpp` 文件中，把输入写到临时输入文件中。
4. 调用编译模块，用 `fork` 创建子进程。
5. 子进程中通过 `execlp("g++", ...)` 执行编译命令。
6. 父进程通过 `waitpid` 等待编译结束。
7. 如果编译成功，再次 `fork` 创建运行子进程执行生成的程序。
8. 父进程等待运行结束，并读取程序输出结果。

对应文件：

- `/root/ONLINEJUDGE/compile_serve/compile_server.cc`
- `/root/ONLINEJUDGE/compile_serve/compile_run.hpp`
- `/root/ONLINEJUDGE/compile_serve/compiler.hpp`
- `/root/ONLINEJUDGE/compile_serve/runner.hpp`

### 为什么要用 `fork + execlp + waitpid`

- `fork` 用来创建子进程，避免主服务直接执行用户代码。
- `execlp` 用来让子进程替换成 `g++` 编译器进程，或者替换成用户程序进程。
- `waitpid` 用来让父进程等待子进程结束，并获取编译或运行状态。

### 你可以怎么讲

> 编译运行模块本质上是一个小型沙箱执行流程。我会先把用户提交的代码写到临时文件里，再通过 `fork + execlp` 调用 `g++` 编译，父进程用 `waitpid` 等待结果。编译成功后，再用同样方式执行生成的可执行程序，并收集运行结果返回给上层服务。

### 面试官可能追问

#### 为什么不直接 `system("g++ ...")`？

建议回答：

- `system` 可控性比较差。
- `fork + execlp` 能更清晰地区分父子进程逻辑。
- 方便做输入输出重定向、资源限制和状态回收。

## 亮点 4：利用 `dup2` 将用户程序的标准输入、标准输出和标准错误重定向到文件，统一收集编译错误、运行结果与异常信息并返回前端

### 实现思路

在运行用户程序前，先打开几个临时文件：

- 输入文件：作为程序标准输入
- 输出文件：保存程序标准输出
- 错误文件：保存程序标准错误

然后在子进程里调用：

- `dup2(stdin_fd, STDIN_FILENO)`
- `dup2(stdout_fd, STDOUT_FILENO)`
- `dup2(stderr_fd, STDERR_FILENO)`

这样用户程序运行时：

- 从输入文件读取数据
- 输出写到 stdout 文件
- 错误信息写到 stderr 文件

执行结束后，父进程把这些文件内容读出来，封装成 JSON 返回。

### 这样做的好处

- 不需要和用户程序做复杂的管道通信。
- 输出和错误分离，便于展示和定位问题。
- 编译错误、运行错误、正常输出都可以统一处理。

### 你可以怎么讲

> 为了收集用户程序的输入输出，我在子进程执行前用 `dup2` 做了标准输入输出重定向。这样程序读写的其实是临时文件，执行结束后父进程直接把这些文件读出来，就能拿到运行结果和错误信息。

### 面试官可能追问

#### 为什么标准错误也要单独重定向？

可以答：

- 因为用户程序运行时可能有崩溃信息或错误输出。
- 分开保存更容易区分正常输出和异常信息。

## 亮点 5：基于 `setrlimit` 对用户程序设置 CPU 和内存限制，拦截超时、非法访存等异常场景

### 实现思路

用户代码执行前，在子进程里设置资源限制：

- `RLIMIT_CPU`：限制 CPU 使用时间
- `RLIMIT_AS`：限制进程可用虚拟内存

如果用户程序：

- 死循环占用 CPU
- 内存申请过大
- 访问非法地址

那么程序可能会被系统信号终止，比如：

- `SIGXCPU`
- `SIGSEGV`
- `SIGABRT`
- `SIGFPE`

父进程在 `waitpid` 返回后，使用：

- `WIFEXITED`
- `WIFSIGNALED`
- `WTERMSIG`

判断子进程是正常退出还是被信号杀死，再把对应状态转换成更易读的描述返回前端。

### 你可以怎么讲

> 因为 OJ 需要运行用户提交的代码，所以我在运行阶段用 `setrlimit` 对 CPU 时间和内存做了限制，避免死循环或者恶意申请大量内存影响服务。运行结束后，父进程会根据退出状态或信号类型判断是正常结束、超时还是非法内存访问。

### 面试官可能追问

#### 为什么限制的是 `RLIMIT_AS`？

可以答：

- `RLIMIT_AS` 限制的是进程地址空间大小，实现上比较直接。
- 对这个项目来说足够完成基础内存限制功能。

#### 这就完全安全了吗？

要实话实说：

> 还不算完全安全。这个项目主要做了基础资源限制，但没有做到容器级隔离、系统调用白名单、文件系统隔离这些更强的安全措施，所以它更适合作为学习性质的 OJ 项目。

这个回答很稳。

## 亮点 6：实现简单负载均衡模块，维护多台编译机在线状态与负载信息，优先选择低负载机器处理请求，并在后端故障时自动摘除离线节点

### 实现思路

这个功能主要在 `oj_control.hpp` 里的 `LoadBalance` 类中实现。

#### 第一步：加载编译机配置

程序启动时会读取配置文件：

- `service_machine.conf`

配置里写多台编译机的地址，例如：

- `127.0.0.1:8080`
- `192.168.x.x:8080`

读取成功后，把机器保存到内存中，并初始化为在线状态。

#### 第二步：维护机器负载

每台机器有一个 `load_` 字段，表示当前正在处理多少请求。

当分发请求给某台机器时：

- 先给它 `IncLoad()`

请求结束后：

- 再 `DecLoad()`

#### 第三步：选择低负载机器

每次收到判题请求时：

- 从在线机器列表中挑选当前 `load` 最小的一台
- 优先把请求发给负载更低的编译机

#### 第四步：失败摘除

如果某台编译机请求失败：

- 说明它可能挂了或网络不可达
- 就把它从在线列表移到离线列表
- 下一次分发时不再选它

### 你可以怎么讲

> 为了避免所有判题都集中在一台机器上，我写了一个简单的负载均衡模块。程序启动时会加载编译机配置，每次判题都优先选择当前负载最小的机器处理；如果请求某台编译机失败，就把它先摘到离线列表，避免后续继续分发到故障节点。

### 面试官可能追问

#### 这属于什么负载均衡策略？

可以答：

- 本质上是最小负载优先。
- 不算复杂算法，核心是根据当前正在处理的请求数做简单选择。

#### 有恢复机制吗？

根据当前代码，建议实话实说：

> 目前实现了失败摘除，但还没有做自动恢复探活，离线机器恢复上线这一块还可以继续完善。

## 三、题库模块是怎么实现的

### 1. 题库数据存储方式

题库没有用数据库，而是基于文件组织：

- `questions.list`：记录题号、标题、难度、CPU 限制、内存限制
- 每道题目录下有：
  - `desc.txt`
  - `header.cpp`
  - `tail.cpp`

### 2. 为什么这样设计

- 对校招项目来说实现简单。
- 题目数据可直接编辑和维护。
- 题目模板天然适合 OJ 场景。

### 3. header 和 tail 是干什么的

这是这个项目比较有特点的一点。

一般用户在网页上只需要补全函数体，或者只写核心逻辑，不需要自己写完整框架。

所以系统会：

- 在 `header.cpp` 中预置头文件、类定义、函数声明
- 在 `tail.cpp` 中补上测试入口或收尾代码
- 用户提交的代码夹在中间

最后拼成一份完整可编译源码。

### 你可以怎么讲

> 题库部分我采用了文件化管理。每道题除了题面描述，还有 `header.cpp` 和 `tail.cpp` 两部分模板，用户提交代码时会和这两个模板拼接成完整源码。这样既方便控制题目输入输出格式，也能让用户更聚焦在核心算法实现上。

## 四、判题接口是怎么实现的

### 1. `POST /judge/{number}` 做了什么

这个接口可以拆成以下步骤：

1. 根据题号查题目是否存在。
2. 解析请求体 JSON，取出用户代码和输入数据。
3. 把 `header + user_code + tail` 拼成完整代码。
4. 把完整代码、输入、CPU 限制、内存限制组装成新的 JSON。
5. 如果配置了远程编译机，就通过负载均衡转发给编译服务。
6. 如果没有配置远程机器，就本地直接调用 `CompileAndRun::Start()`。
7. 把返回结果写到 HTTP 响应中。

### 2. 为什么支持本地执行和远程执行两种模式

可以答：

- 本地执行适合开发和单机调试。
- 远程执行适合后续拆分部署和多机扩展。
- 这样开发和部署都比较灵活。

## 五、状态码和结果返回是怎么设计的

### 1. 返回内容包括什么

编译或运行结束后，系统会返回 JSON，常见字段包括：

- `status`
- `reason`
- `stdout`
- `stderr`
- `compiler_error`

### 2. 为什么这样设计

- `status` 方便程序判断。
- `reason` 方便人看。
- `stdout` 展示程序正常输出。
- `stderr` 展示运行时报错信息。
- `compiler_error` 单独展示编译错误，便于用户修改代码。

### 3. 你可以怎么讲

> 我把判题结果统一封装成 JSON 返回，里面既有机器可判断的状态码，也有用户可读的错误原因，同时保留编译错误、标准输出和标准错误，方便前端展示和问题定位。

## 六、面试时适合讲的项目难点

### 难点 1：把完整判题链路串起来

> 第二个难点是用户代码执行的风险控制。因为用户代码可能死循环、越界访问或者恶意占用资源，所以我用 `setrlimit` 做了 CPU 和内存限制，并通过退出状态和信号来区分不同异常场景。

### 难点 3：编译服务拆分和负载分发

你可以这样讲：

> 第三个难点是把编译执行能力从主服务里拆出去，并设计一个简单的分发逻辑。虽然当前负载均衡策略比较基础，但已经能体现服务拆分和后端调度的思路。

## 七、面试时怎么把项目介绍扩展成 3 分钟

你可以直接按下面这段说：

> 我做的是一个基于 C++ 的在线 OJ 判题系统，项目分成 OJ 主服务和编译运行服务两部分。主服务主要负责题库展示、题目详情和代码提交入口，编译服务负责真正执行用户提交的代码。  
>  
> 在实现上，我用 `cpp-httplib` 提供 HTTP 接口，并在主服务里做了 `Model + Control + View` 分层。题库数据是基于文件加载的，每道题包含题面、代码模板和资源限制。用户提交代码后，系统会把用户代码和题目模板拼接成完整源码，再把代码、输入、CPU 和内存限制封装成 JSON 发送给编译服务。  
>  
> 编译服务内部会先生成临时文件，再通过 `fork + execlp + waitpid` 完成编译和运行，并用 `dup2` 做标准输入输出重定向，用 `setrlimit` 限制 CPU 和内存资源，最后把编译错误、运行输出和异常状态封装成 JSON 返回。除此之外，我还实现了一个简单的负载均衡模块，可以根据编译机当前负载选择后端节点，并在节点不可用时做摘除。

## 八、项目不足怎么说更稳

这个也要提前准备，不然面试官问到容易卡住。

建议回答：

- 当前题库数据还是文件存储，没有接数据库。
- 负载均衡只有简单的最小负载策略，没有做自动恢复和健康检查。
- 用户代码执行虽然做了资源限制，但还没有做到容器级隔离。
- 判题结果目前更多是编译运行结果，没有做标准答案比对和完整测试用例系统。

这几个不足都很真实，也不伤项目质量。

7. 负载均衡为什么能按最小负载选机器。
8. 项目目前的边界和不足是什么。

## 十、一句话兜底模板

如果面试官追问到特别细，而你一时答不全，可以这样接：

> 这个点我项目里已经做了基础实现，重点是先把主流程跑通。更完整的方案，比如更强隔离、更复杂负载均衡和自动恢复，目前我有了解思路，但还没有在这个项目里继续展开。

这句话对校招实习非常好用。
