# Problem Notes

这个文档用于沉淀项目里实际踩过的问题，方便后续复盘、面试表达和持续补充。

## 使用方式

- 每次新增问题时，按日期新开一个小节。
- 每个问题尽量写清楚现象、根因、解决方法和面试可表达点。
- 优先记录有工程价值的问题，不要只记“手误”，要记“为什么会出问题，以及怎么避免再次发生”。

## 记录模板

### [日期] 问题标题

**分类**

编译期 / 链接期 / 路径与环境 / 进程控制 / 文件系统 / 工程设计 / 其他

**现象**

写清楚报错、异常行为或错误结果。

**根因**

说明问题为什么发生，尽量说到底层机制，不只描述表面现象。

**解决方法**

写清楚改了什么，以及为什么这样改。

**面试表达**

用 2 到 4 句话总结这个问题的工程价值和你的处理思路。

## 2026-03-26

### 1. `LOG.hpp` 中的字符串初始化和命名空间错误导致编译失败

**分类**

编译期

**现象**

- 工程在编译 `compile_server` 时直接失败。
- 典型报错包括：
  - `conversion from 'char' to non-scalar type 'std::string' requested`
  - `'util' has not been declared`

**根因**

- `std::string message='[';` 试图用字符字面量初始化字符串对象，类型不匹配。
- 时间工具类定义在 `ns_util` 命名空间下，但调用时误写成了 `util::Timeutil::GetTimeStamp()`。
- `#pragma once` 原本还被拼错，头文件保护本身也不可靠。

**解决方法**

- 将字符串初始化改为 `std::string message = "[";`。
- 将时间戳调用改为 `ns_util::Timeutil::GetTimeStamp()`。
- 修正头文件保护为 `#pragma once`。

**面试表达**

这个问题表面是语法错误，但本质上反映了头文件细节不严谨会直接阻断整个构建链路。我的处理方式不是只改报错行，而是顺着编译器输出把类型、命名空间和头文件保护一起校正，先确保工程能稳定进入下一阶段排查。

### 2. 头文件中定义普通函数，`#pragma once` 不能替代 `inline`

**分类**

链接期 / 工程设计

**现象**

- 当前工程只有一个源文件时未必暴露问题。
- 但如果后续多个 `.cpp` 都包含 `LOG.hpp`，链接阶段可能出现重复定义。

**根因**

- `#pragma once` 只防止同一个头文件在单个编译单元内被重复包含。
- 它不能防止同一个头文件里的函数定义在多个编译单元中各自生成一份。
- `Log(...)` 这种直接定义在头文件中的普通函数，如果没有 `inline`，后续扩展工程时容易触发 ODR 相关的重复定义问题。

**解决方法**

- 将头文件中的 `Log(...)` 定义改为 `inline`。
- 同时把宏改为显式调用 `ns_log::Log(...)`，减少名字冲突风险。

**面试表达**

这个问题的价值不在于“会不会写 `inline`”，而在于是否理解 `#pragma once` 和 `inline` 解决的是两个不同层面的事情。前者解决重复包含，后者解决多编译单元下的函数定义合法性，这是 C++ 头文件设计里的基本工程意识。

### 3. `./temp/` 相对路径依赖当前工作目录，导致运行环境不稳定

**分类**

路径与环境

**现象**

- 从仓库根目录运行程序和从 `compile_serve` 目录运行程序时，临时文件路径不一致。
- 程序可能找不到源文件、错误文件或生成的可执行文件。

**根因**

- `./temp/` 不是相对源码文件解析，而是相对进程启动时的当前工作目录解析。
- 原实现默认认为 `./temp/` 永远指向 `compile_serve/temp/`，这个假设不成立。

**解决方法**

- 将临时目录路径收敛到统一的路径解析逻辑中。
- 当前实现优先尝试 `./temp/`，不存在时再尝试 `./compile_serve/temp/`，兼容两种常见启动方式。
- 更长期的做法是显式传入路径根目录，避免路径依赖运行位置。

**面试表达**

这类问题不是语法层面的，而是运行环境耦合问题。我的经验是只要程序用相对路径，就必须明确它相对的是谁，否则一旦换启动目录、换脚本或换部署方式，程序就会变得非常脆弱。

### 4. 只靠可执行文件是否存在来判断编译成功，会出现“假成功”

**分类**

进程控制 / 文件系统

**现象**

- 某次编译失败后，程序仍可能判断“编译成功”。
- 尤其是在上一次成功编译生成的旧 `.exe` 文件还存在时，这个问题更容易出现。

**根因**

- 原逻辑只检查目标可执行文件是否存在，没有结合这次 `g++` 子进程的真实退出状态。
- 旧产物残留会污染本次判定结果。

**解决方法**

- 编译前先删除旧的可执行文件。
- `waitpid` 后检查子进程是否正常退出，并确认退出码为 `0`。
- 最后再辅以检查目标文件是否生成，作为结果确认。

**面试表达**

这个问题体现的是“过程状态”和“最终产物”不能混为一谈。工程上判断一次任务是否成功，优先看本次执行的退出状态，再看产物是否符合预期，否则很容易被历史残留文件误导。

### 5. `dup2` 和错误重定向只写主路径，不处理失败分支会影响排障

**分类**

进程控制 / 文件描述符

**现象**

- 标准错误重定向如果失败，编译错误信息可能不会落到 `.err` 文件中。
- 文件描述符如果不关闭，会留下不必要的资源占用。

**根因**

- 系统调用只写了 happy path，没有完整处理失败场景和资源生命周期。
- `open`、`dup2`、`close` 这类接口都需要成对考虑成功路径和异常路径。

**解决方法**

- 检查 `open` 和 `dup2` 的返回值。
- `dup2` 成功后关闭原始文件描述符。
- 出错时记录日志并及时退出，避免错误状态继续向后传递。

**面试表达**

这类问题比较能体现系统编程意识。写 `fork/exec/dup2` 不只是把主流程跑通，更重要的是保证失败时还能留下可定位的信息，并且不引入新的资源管理问题。

### 6. 默认输入文件名与实际项目资源不一致，导致工具不可用

**分类**

工程设计 / 接口设计

**现象**

- 主程序默认编译 `test`，但项目里实际存在的是 `code.cpp`。
- 程序行为和当前目录结构不一致，导致默认执行失败。

**根因**

- 默认参数脱离了实际资源布局。
- 工具入口被硬编码，复用性差。

**解决方法**

- 将默认文件名改为当前实际存在的 `code`。
- 同时支持命令行参数，让编译目标可以动态指定。

**面试表达**

默认值应该服务当前场景，但不能把程序写死。我的处理方式是保留合理默认值，同时把入口参数化，这样工具既能直接用，也能扩展到更多输入场景。

### 7. 排查问题时需要先分层，不要把多个问题混成一个问题

**分类**

调试方法 / 工程实践

**现象**

- 最初看上去只是一个编译错误，继续排查后发现同时叠加了路径问题和成功判定问题。
- 如果只盯着第一个症状，很容易修掉一个点后又被下一个问题卡住。

**根因**

- 编译期、启动期、运行期的问题混在一起出现时，表面症状会相互遮挡。
- 如果不按阶段拆分，定位效率会很低。

**解决方法**

- 先跑构建，确认编译期错误。
- 再检查依赖文件和路径约定，确认环境层问题。
- 最后实际运行并校验产物、退出码和日志，验证链路闭环。

**面试表达**

这次排查让我进一步确认了一件事：复杂问题往往不是一个问题，而是多个层次的问题叠加。有效的方法不是一上来猜结论，而是按编译、启动、运行三个阶段拆开验证，逐层收敛。

### 8. 编译前没有真正删除旧产物时，失败编译可能留下“脏可执行文件”

**分类**

进程控制 / 文件系统

**现象**

- 某个文件上一次编译成功后生成了 `.exe`。
- 本次把源代码改错再重新编译，如果旧的 `.exe` 没被清理干净，后续流程仍可能误用旧二进制。
- 这类问题在“编译失败后又直接运行”或者“只看产物是否存在”时尤其隐蔽。

**根因**

- 编译前清理旧产物的动作如果没有真正执行，历史成功结果就会污染本次任务。
- 这类问题本质上不是编译器的问题，而是任务前置清理步骤失效，导致“本次状态”和“历史状态”混在一起。

**解决方法**

- 在编译开始前显式调用 `::unlink(exe_file.c_str())` 删除旧的可执行文件。
- 编译失败时只信任当前 `g++` 子进程的退出状态和本次生成的错误文件，不再依赖历史残留产物。
- 验证时可以先手工放一个旧 `.exe`，再构造语法错误源码，确认失败后旧产物确实被删掉。

**面试表达**

这个问题很适合体现工程上的“状态隔离”意识。一次编译任务的结果必须只由本次执行决定，不能被上一次成功构建留下的文件污染，所以我会在任务开始前主动清理旧产物，再结合退出码和错误文件判断本次是否成功。

### 9. 运行阶段直接阻塞等待子进程，会被死循环程序拖死

**分类**

进程控制

**现象**

- 用户程序如果写成死循环，父进程会一直卡在 `waitpid(..., 0)`。
- 工具表面上没有崩，但调用方会长期拿不到返回结果。
- 在线判题或批量执行场景里，这会直接拖垮后续任务调度。

**根因**

- `waitpid(..., 0)` 是阻塞等待，默认假设子进程最终一定会自己结束。
- 但用户程序可能出现死循环、卡死、长时间阻塞等情况，这个假设不成立。
- 只要没有超时控制，父进程就失去了对运行时长的主动权。

**解决方法**

- 改成 `waitpid(pid, &status, WNOHANG)` 轮询检查子进程状态。
- 父进程每隔很短时间检查一次，超过设定时间上限后主动 `kill(pid, SIGKILL)`。
- 最后再 `waitpid` 一次把被杀掉的子进程回收，避免僵尸进程。

**面试表达**

这类问题体现的是“运行控制权”不能完全交给用户程序。我的处理方式不是一味阻塞等待，而是让父进程保留超时回收能力，这样即使用户代码死循环，服务本身也不会被无限卡住。

### 10. 用保留退出码区分内部错误，可能和用户程序真实返回值冲突

**分类**

进程控制 / 接口设计

**现象**

- 运行器内部如果约定某个特殊退出码表示“系统错误”，看起来实现简单。
- 但当用户程序自己刚好 `return` 了这个值时，父进程会把正常的用户退出误判成运行器错误。
- 例如用户程序返回 `250` 时，原逻辑会把它归错类。

**根因**

- 子进程退出码本来就是用户程序和运行器共享的一条通道。
- 如果没有额外的隔离机制，运行器私自占用某个退出码，就会和用户程序的合法返回值空间冲突。
- 这个问题本质上是接口设计冲突，不是单纯的判断分支写错。

**解决方法**

- 在当前简化版本里，不再保留特殊退出码做细分。
- 用户程序只要是非 `0` 正常退出，就统一归类为 `kRunProgramError`。
- 如果以后确实需要区分“运行器内部错误”和“用户程序退出码”，应单独引入更可靠的通信方式，而不是抢占一个返回值。

**面试表达**

这个问题能体现我对接口边界的敏感度。退出码表面上只是一个整数，但它其实是父子进程之间的共享协议；如果运行器随意占用某个值，就会和用户程序的语义打架，所以我会优先保证协议不冲突，再考虑是否需要更细粒度的信息传递。
