# OJ 项目面试问答

这份文档按当前项目实现整理，回答口径尽量贴合代码事实，不把项目吹成生产级安全 OJ。

## 30 秒项目介绍

我做的是一个基于 C++ 的在线判题系统。整体分成两个服务：`oj_server` 负责题库展示、题目详情和提交入口，`compile_server` 负责代码编译与运行。用户提交代码后，系统会把题目的 `header + 用户代码 + tail` 拼成完整程序，再通过 `fork/exec/waitpid` 完成编译运行，并结合 `setrlimit` 做 CPU 和内存限制；如果配置了多台编译机，还支持基于最小负载的请求分发和故障摘除。

## 项目常见问题

### 1. 为什么要做这个项目？

我想做一个能把 C++、网络编程、进程控制和工程组织串起来的项目。OJ 这个场景比较完整，从前端提交、后端调度到编译运行结果返回，链路很闭环，比较适合体现后端工程能力。

### 2. 这个项目为什么和其他同学做的项目看起来类似？

OJ 本身就是经典练手项目，题材相似很正常。真正拉开差距的是实现深度。我的项目不只是本地跑代码，而是做了主服务和编译服务拆分，支持题库模板拼接、远端执行、简单负载均衡和失败节点摘除，这些是我重点做的工程化部分。

### 3. 介绍一下这个项目的整体流程和架构设计？

整体是两层架构：

- `oj_server`：负责 HTTP 路由、题库加载、页面渲染和判题请求组织。
- `compile_server`：负责真正的编译和运行。

用户请求流程如下：

1. 前端访问题目列表或题目详情，由 `oj_server` 从题库模型中读取题目信息并渲染页面。
2. 用户在页面中提交代码，请求会打到 `POST /judge/{number}`。
3. `Control::Judge` 会先根据题号查题，再解析前端提交的 JSON。
4. 系统把代码拼装成 `header + user_code + tail`，并附带 `input`、`cpu_limit`、`mem_limit`。
5. 如果配置了编译机，就走 `LoadBalance::Dispatch` 发给远端 `compile_server`；否则本地直接调用 `CompileAndRun::Start`。
6. 编译运行结束后返回 JSON，包括 `status`、`reason`、`stdout`、`stderr`、`compiler_error`。
7. 前端根据返回结果展示“通过 / 未通过 / 编译错误 / 运行异常”等状态。

### 4. 项目最终成果如何？

目前项目已经实现了这些核心能力：

- 题库加载与展示
- 题目详情页
- 模板化判题
- 本地编译运行
- 远端编译服务调用
- 简单负载均衡和失败节点摘除

主流程已经是闭环的，项目可以正常编译、启动和演示。它已经是一个可运行的在线判题系统原型。

### 5. 你从这个项目里学到了什么？

我主要有三点收获：

- 把操作系统相关知识落地到工程里，比如 `fork/exec/waitpid`、文件重定向、信号和资源限制。
- 理解了模块拆分的重要性，比如题库模型、控制层、编译运行层和负载均衡层要清晰分开。
- 学到了很多工程上的细节问题，像相对路径、临时文件命名、编译成功判定、进程退出状态和异常分支处理，这些比单纯写业务逻辑更锻炼工程能力。

### 6. 你在项目中用了哪些技术？为什么这样选型？

我使用的核心技术包括：

- C++：因为这个项目和进程控制、资源限制、文件描述符等底层能力结合得比较紧。
- `cpp-httplib`：头文件式库，轻量，接入成本低，适合个人项目快速搭建 HTTP 服务。
- `jsoncpp`：用于请求和响应 JSON 的解析与序列化，够直接。
- 文件系统题库：因为项目规模不大，题目数据结构比较固定，文件组织比数据库更轻量。
- `fork/exec/waitpid`：实现编译和运行的子进程控制。
- `setrlimit`：控制用户程序的 CPU 和内存资源使用。

选择这些技术的原因是：实现成本低、和项目规模匹配、便于展示系统编程能力。相比更重的方案，比如数据库、gRPC 或完整 Web 框架，这套选型更适合这个阶段的项目。

### 7. 你对比过哪些其他技术方案？

做过一些取舍思考：

- 单机本地判题 vs 主服务和编译服务拆分：我最终做了拆分。因为判题执行风险高、资源消耗也更大，拆开以后更容易隔离风险，也方便后续扩容多台编译机。
- 多线程执行用户代码 vs 多进程执行用户代码：我选多进程。因为用户代码是不可信的，进程模型更适合配合 `fork/exec/waitpid`、资源限制和异常回收，隔离性比线程更好。
- 文件题库 vs 数据库：我选文件题库。因为当前题目数量不大、写操作很少、结构也比较固定，直接按目录组织 `desc/header/tail` 更轻量，开发效率也更高。
- HTTP + JSON vs 基于 TCP 自定义应用层协议：我选 HTTP + JSON。原因是开发门槛更低、跨主机调试方便、和浏览器前端天然兼容，也便于后续直接抓包和排查问题。
- 轻量级 HTTP 库 vs 更重的网络框架：我选 `cpp-httplib`。因为这个项目接口数量不多，核心难点不在 Web 框架本身，而在判题链路和进程控制，轻量方案更符合当前阶段。

### 8. 你在项目中扮演了什么角色？

如果按个人项目来答，我是独立完成。需求拆解、架构设计、题库模型、判题链路、编译运行、负载均衡、页面交互和最终联调都是我自己完成的。

### 9. 你负责了哪些模块？

我主要负责这些模块：

- `Model`：题库数据加载与查询
- `View`：题目页面和结果页面渲染
- `Control::Judge`：判题请求总入口
- `CompileAndRun`：编译和运行主流程
- `Compiler` / `Runner`：进程级编译运行
- `LoadBalance`：编译机分发与故障摘除

### 10. 你是如何平衡技术实现和项目时间线的？

我的策略是先打通主链路，再逐步增强：

1. 先做单机版 OJ，确保题库、提交、编译运行闭环。
2. 再补模板化判题和错误处理。
3. 最后再拆分 `oj_server` 和 `compile_server`，做远端执行和负载均衡。

这样做的好处是每个阶段都有一个可以运行的版本，不会因为一开始追求复杂架构而一直没有可交付结果。

### 11. 你在项目中遇到了哪些挑战？怎么解决的？

我遇到过几个比较典型的问题：

- 临时文件名冲突：用“微秒时间戳 + 进程号 + 原子自增序号”解决唯一性问题。
- 相对路径不稳定：把题库路径和临时目录路径做成候选路径查找，减少运行目录依赖。
- 编译成功误判：不能只看产物文件存在，还要结合 `waitpid` 返回状态和退出码判断。
- 运行错误定位困难：把标准输入、标准输出、标准错误和编译错误都落到独立文件，便于排查。

### 12. 你是怎么测试和部署这个项目的？

测试方面，目前以功能测试和联调测试为主，包括：

- 题库能否正确加载
- 编译错误是否能正确返回
- 正常运行是否能返回输出
- 运行异常是否能区分段错误、超时等情况
- 编译机不可达时是否会摘除节点并返回错误

部署方式是：

1. 启动一个或多个 `compile_server`
2. 在 `service_machine.conf` 里配置编译机 `ip:port`
3. 启动 `oj_server`
4. 如果没有配置远端编译机，`oj_server` 会退回本地 `CompileAndRun` 模式

### 13. 项目性能如何？做过哪些优化？

这个项目的主要瓶颈不在普通业务逻辑，而是在代码编译和子进程创建上。当前做过的优化包括：

- 主服务和编译服务拆分，便于横向扩展编译能力
- 支持多编译机最小负载分发
- 失败节点自动摘除，减少无效重试
- 使用轻量 HTTP 库，避免框架开销过大

如果后续继续优化，我会考虑编译机健康检查、编译缓存、任务队列和更强的并发控制。

### 14. 你是如何设计项目以支持未来扩展的？

我做了几件事来支持扩展：

- 模块分层比较清晰：题库、控制层、视图、编译器、运行器、负载均衡各自职责明确
- 主服务和编译服务拆开，未来可以继续增加编译节点
- 服务之间使用 JSON over HTTP，扩展新节点和新字段比较方便
- 题库使用文件结构组织，新增题目成本低

### 15. 你们是如何保证代码可维护性的？

目前主要靠这几件事：

- 模块职责清晰，减少跨层耦合
- 对关键函数加职责注释，便于后续阅读
- 把常见问题沉淀到 `problem_notes.md`
- 接口设计保持简单，比如题库读取接口和编译执行输入输出格式都比较明确

如果继续完善，我会补单元测试、接口测试和更系统的文档。

## 技术细节追问

### 16. 临时文件的命名唯一性如何解决？

我在 `FileUtil::UniqFileName()` 里用三个维度组合：

- 微秒级时间戳
- 当前进程 PID
- 原子自增序号

这样即使同一时刻多个请求并发，也能避免源文件、输入文件、输出文件和可执行文件之间发生命名冲突。

### 17. 如何判定编译成功？如何判定运行成功？

编译成功不是只看 `.exe` 文件是否存在，而是同时看两个条件：

- `waitpid` 返回的子进程状态正常退出，且退出码为 0
- 目标可执行文件确实生成

运行成功则是：

- `waitpid` 后满足 `WIFEXITED(status)`
- `WEXITSTATUS(status) == 0`

如果 `WIFSIGNALED(status)` 成立，就说明是被信号中断，比如 `SIGSEGV`、`SIGFPE`、`SIGXCPU` 这些运行异常。

### 18. `oj_server` 和 `compile_server` 能否部署在不同主机上？

可以，这就是当前架构支持的方式。`oj_server` 会从配置文件中读取 `ip:port`，再通过 HTTP 把判题请求发给远端的 `compile_server`。只要网络可达，就可以跨机器部署。

### 19. `oj_server` 和 `compile_server` 之间的数据交互格式是怎样的？

使用 JSON。

请求大致包含：

- `code`
- `input`
- `cpu_limit`
- `mem_limit`

响应大致包含：

- `status`
- `reason`
- `stdout`
- `stderr`
- `compiler_error`

这种格式的好处是结构清晰、易于调试、前后端都容易处理。

### 20. 介绍一下 HTTP 报文格式？

HTTP 请求报文一般由四部分组成：

- 请求行：例如 `POST /judge/1000 HTTP/1.1`
- 请求头：例如 `Host`、`Content-Type`、`Content-Length`
- 空行
- 请求体：这里一般放 JSON

HTTP 响应报文也类似：

- 状态行：例如 `HTTP/1.1 200 OK`
- 响应头
- 空行
- 响应体：这里一般是 JSON 或 HTML

### 21. 为什么使用 HTTP 作为服务器之间的通信协议？

主要有几个原因：

- 简单直接，开发成本低
- 跨主机通信天然支持
- 和浏览器前端天然兼容
- 调试方便，抓包和测试都容易
- 当前项目规模不大，没有必要上更重的 RPC 框架

### 22. `fork` 的工作原理是什么？

`fork` 会创建一个子进程。父子进程在逻辑上有相同的地址空间内容，但通常底层使用写时拷贝来避免真的立刻完整复制。调用 `fork` 后会返回两次：

- 在父进程里返回子进程 PID
- 在子进程里返回 0

这样父进程就可以等待子进程结果，子进程则可以继续 `exec` 去执行新的程序。

### 23. 为什么这里使用多进程编程？

因为用户提交的代码是不可信的。使用多进程的好处是：

- 编译和运行可以与主服务隔离
- 可以结合 `setrlimit` 对子进程做资源限制
- 可以通过 `waitpid` 获取退出状态和信号
- 子进程崩溃不会直接把主服务带崩

相比线程模型，多进程更适合这种“执行不可信代码”的场景。

### 24. HTTP 服务器是如何搭建的？

我用的是 `cpp-httplib`。实现方式比较直接：

1. 创建 `httplib::Server`
2. 注册 `Get` 和 `Post` 路由
3. 路由中调用业务逻辑
4. 最后 `listen("0.0.0.0", port)` 启动服务

比如：

- `oj_server` 里有 `/all_questions`、`/question/{number}`、`/judge/{number}`
- `compile_server` 里有 `/compile_and_run`

### 25. 为什么要把 `oj_server` 和 `compile_server` 拆开？

因为两个模块职责不同：

- `oj_server` 更偏业务入口和页面服务
- `compile_server` 更偏高风险执行环境

拆开之后有几个好处：

- 更容易单独部署和扩容
- 主服务不直接承担所有编译执行压力
- 有利于后续做多编译机调度
- 安全边界更清晰

### 26. 负载均衡是怎么做的？

当前是比较简单的最小负载策略：

- 每台机器维护一个 `load_` 计数，表示当前正在处理的任务数
- 调度时遍历在线机器，挑负载最小的那台
- 请求发出前先 `IncLoad()`
- 请求结束后 `DecLoad()`
- 如果请求失败，就把该节点从在线列表移到离线列表

这套方案简单有效，适合当前项目规模。

## 代码与设计追问

### 27. `Question` 对象是干什么的？

`Question` 表示一道题在内存中的完整数据结构，包含：

- 题号
- 标题
- 难度
- CPU 限制
- 内存限制
- 题目描述
- `header`
- `tail`

它既服务于题目展示，也服务于判题逻辑，因为判题时需要把 `header + 用户代码 + tail` 拼成完整程序。

### 28. `Judge` 的核心逻辑是什么？

`Judge` 是用户提交代码之后的总入口。它会：

1. 根据题号取题
2. 解析前端提交的 JSON
3. 拼接题目模板和用户代码
4. 组装编译执行参数
5. 选择远端或本地执行路径
6. 返回判题结果

它自己不直接编译，而是负责组织请求并调度到 `CompileAndRun` 或远端编译服务。

### 29. `ParseMachineLine` 和 `LoadConf` 是做什么的？

- `ParseMachineLine`：负责把一行 `ip:port` 配置解析成主机地址和端口
- `LoadConf`：负责读取配置文件、逐行解析、创建 `Machine` 对象，并把机器加入在线列表

两者配合后，`LoadBalance` 才能知道有哪些编译节点可用。

### 30. 为什么 `Machine` 要配合智能指针使用？

因为 `Machine` 里有 `std::mutex`，对象不适合按值拷贝；同时系统里多个位置可能同时引用同一台机器。用智能指针管理后：

- 生命周期自动托管
- 不需要手写 `new/delete`
- 复制的是指针所有权，不是 `Machine` 对象本体
- 不会去复制里面的 `mutex`

## 风险和不足

### 31. 这个项目最大的缺陷是什么？

最大的缺陷是安全隔离还不够强。当前主要是：

- 使用 `setrlimit` 限制 CPU 和内存
- 使用独立子进程执行用户代码

但它还不是生产级沙箱，没有做到容器隔离、降权用户、系统调用白名单、网络隔离、进程数限制、输出大小限制等更严格的安全措施。

### 32. 这个项目能不能写进简历？

可以，而且是可以作为一个比较完整的个人后端项目写进去的。因为它已经覆盖了：

- HTTP 服务开发
- 模型与控制层设计
- 进程控制
- 编译运行链路
- 简单负载均衡
- 服务拆分

但简历表述要准确，建议写成“在线判题系统原型 / 教学型 OJ / 支持远端执行和简易负载均衡的在线判题系统”，不要写成“生产级安全 OJ 平台”。

### 33. 如果继续做下一版，你会优先补什么？

我会优先补三件事：

- 更强的沙箱与隔离能力
- 把“通过 / 未通过”的 verdict 完全下沉到后端，不依赖前端解析输出
- 为编译机增加健康检查和自动恢复能力，而不是只做失败摘除
